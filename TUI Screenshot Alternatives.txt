For a TUI, the best options are usually not raw screenshots alone, because screenshots show what happened but not always how you got there. The sweet spot is one of these:

1. Record terminal sessions as text, then render frames/screenshots from them
This is usually the nicest option for a coding agent.

Good tools:

asciinema — records terminal interactions as a replayable cast

VHS — lets you script terminal demos and export them as GIF/video

termtosvg — records a session and exports SVGs/animations

script / ttyrec — lower-level session recorders

Why this is great:

reproducible

searchable

easy to regenerate after UI changes

much better than taking 37 screenshots like a caffeinated raccoon

A very practical workflow is:

record a flow with asciinema rec

replay or convert it

extract key frames for “screen A -> screen B -> modal -> confirm -> success”

give those frames plus a tiny text note for each step to the coding agent

Example:

asciinema rec flow.cast
# do the interaction
exit
Then either share the .cast itself, or render it with another tool.

2. Generate clean scripted demos
If you want something polished, use VHS.

With VHS you define the flow as a script, like:

Output demo.gif

Set Shell "bash"
Set FontSize 16
Set Width 1200
Set Height 700

Type "my-tui-app"
Enter
Sleep 1s
Type "j"
Sleep 500ms
Type "enter"
Sleep 1s
This is fantastic because your “design flow” becomes versionable. The agent can read both:

the scripted interaction

the resulting GIF/screenshots

That is gold for TUI work.

3. Capture screenshots directly from the terminal emulator
If you really want static screenshots, you can do that too.

Depending on terminal:

WezTerm can do window/pane capture through its tooling and config ecosystem

kitty has remote-control features and good rendering behavior

tmux can dump pane contents

normal OS screenshot tools work fine too

For example, with tmux, even if you do not get a literal PNG, you can capture exact pane text:

tmux capture-pane -p > step1.txt
That is often better than an image for an agent, because it sees the real terminal content.

4. Provide text snapshots instead of images
For coding agents, this is often underrated and very effective.

You can save terminal states like:

my-app > step1.txt
my-app --state browse > step2.txt
Or even manually keep mock screens as plain text / markdown code fences:

┌ Project List ──────────────────────┐
│ > API Server                       │
│   Worker Queue                     │
│   Settings                         │
└────────────────────────────────────┘
That gives the agent something deterministic to compare against.

Best combo:

asciinema or VHS for flow capture

a few key PNG/SVG screenshots for visual reference

plain-text screen dumps for exact layout/state

a tiny flow spec like:

land on dashboard

open item details

show success toast

That way the agent gets:

interaction sequence

exact text

visual layout

intended behavior

intended purpose

important regions: header, list, status bar, modal, help footer

constraints: max width, overflow behavior, keyboard shortcuts

- selected row must be high contrast

That makes the agent much smarter than just dumping images and whispering “vibes.”

you already like Rust

you want to prototype fast

Why tmux is a very good Unix backend
For Linux and macOS, tmux gives you a lot of what you want out of the box:

a server/session model

pane/window lifecycle

text capture with capture-pane

automation-friendly command execution

a machine-readable control mode intended for programmatic interaction rather than terminal rendering hacks. 

That means on Unix you can avoid reinventing a PTY supervisor at first. Let tmux own the shell and terminal lifecycle, and let your MCP server act like an orchestrator.

The especially juicy part is control mode. It gives you a text protocol for talking to tmux, which is much cleaner than pretending to be a human mashing keys into a normal terminal window. 

Why Windows needs a different backend
On Windows, the right primitive is ConPTY / pseudoconsole. Microsoft’s pseudoconsole API is specifically designed so a host application can create the terminal session and communicate with the child console app through pipes, which is exactly the kind of plumbing an MCP terminal server wants. 

That makes the Windows backend conceptually:

create pseudoconsole

spawn child process inside it

write input

read output

maintain normalized visible state

expose snapshots to MCP

If you use Node, node-pty already wraps this cross-platform and documents Windows support via ConPTY on Windows 1809+; older winpty support has been removed.

push only the nasty OS-specific bits into native addons when you truly need them

keep the public MCP contract unchanged

So you get:

fast iteration in TS

direct access to platform APIs when needed

no need to commit the whole project to C++/Rust just because one corner is ugly

Why this works well
Your project naturally splits into two layers:

High-level layer in TS

MCP server

tool/resource definitions

flow engine

session lifecycle

snapshot normalization

diffing

orchestration around tmux / PTY

Low-level layer in native code only where needed

Windows-specific ConPTY helpers

process/job-object management

screen buffer or terminal parsing optimizations

oddball platform APIs that are annoying from JS

That is a very healthy architecture. The important part is to treat native code as a backend detail, not as the center of the project.

Good reasons to add native bindings later
Native modules make sense when you hit one of these:

you need direct access to Win32 / ConPTY APIs beyond what existing packages expose

you need better control over process trees, handles, cleanup, signals, job objects

you want faster parsing of huge ANSI streams

you need tighter terminal state tracking than shelling out or using node-pty gives you

you need OS features that are awkward through child-process wrappers

That is much better than prematurely writing the whole MCP in C++ and then spending your life hand-assembling JSON like a medieval blacksmith.

I would still avoid starting with custom native code
Not because it is bad, but because native modules come with taxes:

build tooling

ABI compatibility concerns

Node version friction

platform packaging pain

CI complexity across Linux/macOS/Windows

So I would start with:

TypeScript

tmux backend on Linux/macOS

node-pty on Windows first

only introduce native bindings where there is a real gap

That gives you proof of architecture before you marry the build system.

If you do go native, prefer narrow modules
Do not make one giant addon that “does terminals.”

Much better:

@yourorg/pty-win32

@yourorg/process-tree

@yourorg/ansi-fastpath

Each one should do one ugly thing well.

Then TS stays the conductor, and the native pieces are just the percussion section hitting things loudly when needed.

C++ vs Rust for the native layer
If you are thinking native addons, the usual options are:

C++

easiest fit with Node’s traditional native addon ecosystem

direct access to OS APIs

totally fine for thin wrappers

Rust

safer and often nicer to maintain

can still integrate with Node through addon tooling

stronger choice if the native part grows beyond “thin wrapper”

So:

for small, sharp bindings, C++ is perfectly reasonable

for a larger native subsystem, I would seriously consider Rust instead

Best practical architecture

TypeScript core

MCP protocol

tools/resources

flow engine

snapshot schema

adapter selection by OS

Backend adapters

TmuxBackend

NodePtyBackend

optional NativeWindowsBackend

Optional native helpers

only for capabilities TS cannot do cleanly enough

That way you never paint yourself into a corner.

So yes — your instinct is good
Node/TypeScript is a strong choice partly because it lets you cheat intelligently:

stay productive in TS

drop to native only when justified

keep the system modular

That is usually a much smarter engineering move than starting “pure native” and discovering six weeks later that most of your work is actually orchestration, schemas, and glue.
